# [TIL] 2026-06-03 Gazebo Sensor


## Sensor 타입
gazebo는 많은 센서 모델을 제공한다.
1. 시각 및 공간 인지 센서
    - Camera: 2D 이미지 데이터 생성
    - Depth Camera: 각 픽셀까지의 거리 정보와 함께 제공(3D 가상 공간 인식 시)
    - Lidar / Laser / Ray : 빛을 쏘아 사물과의 거리 측정, 거리 배열/Point Cloud 데이터 반환
    - Logical Camera: 센서 시야 내에 있는 물체의 이름과 정확한 좌표 정보를 텍스트 데이터로 반환
2. 물리적 접촉 및 역학 센서
    - Contact: 로봇의 특정 충돌 영역(Collision)에 다른 물체가 닿았는지 여부(T/F)와 닿은 물체 이름 알려준다.
    - Force / Torque: 로봇의 관절이나 링크에 걸리는 물리적인 힘과 회전력을 3축 기반으로 정밀하게 측정
3. 위치 및 관성 센서
    - IMU(관성 측정 장치): 가속도와 각속도를 측정
    - GPS / NavSat(위성 항법 시스템): 위도, 경도, 고도 데이터 제공(실외 자율주행 로봇, 드론 등에 사용)
4. 기타 특수 센서
    - Sonar(초음파 센서): 음파를 이용해 거리 측정
    - Magnetometer(지자기 센서): 지구 자기장 측정하여 로봇의 절대적인 방위각 제공(like 나침반) 보통 IMU와 함께 결합되어 사용
>> 센서들의 공통점
    1. SDF 파일 안에서 <sensor type="센서 종류"> 태그로 선언
    2. 각 센서마다 고유한 플러그인(Plugin)을 통해 밖으로 데이터를 던져준다.

현실에서는 센서들은 노이즈를 보인다(즉, 이상적인 환경을 관찰할 수 없다.)
기본적으로, Gazebo의 센서들은 이상적인 환경을 관찰한다.

좀 더 현실적인 환경을 제공하기 위해, Gazebo 센서에 의해 만들어진 데이터에 노이즈를 추가할 필요가 있다. (인식(Perception) 코드를 통해 구현)

Gazebo는 아래 센서 타입에 노이즈를 추가할 수 있다.
- Ray(laser/lidar) Noise
    각 빔의 범위에 Gaussian noise를 추가할 수 있다.
    노이즈 값을 샘플링할 가우스 분포(Gaussian distribution)의 평균값(mean), 표준편차(std)를 설정할 수 있다. -> 평균값과 표준편차값 만 조정하면 노이즈 조정 가능
        보통 현실적인 센서를 묘사할 때, 평균을 0으로 설정한다. 원래 거리를 중심으로 앞 뒤로 흔들리는 현실적인 노이즈가 된다.
        만약, 평균을 0.1로 주면, 센서가 항상 실제보다 10cm씩 더 멀리 측정하는 체계적 오차(Bias)를 만들 수 있다.
    노이즈 값은 각 빔마다 독립적으로 샘플링 된다. 
        레이저 빔이 100개 나간다면, 100개의 빔은 각각 서로 다른 오차 값을 가진다.

- Camera noise
각 픽셀에 독립적으로 Gaussian-sampled disturbance를 추가한 output amplifier noise를 모델링한다.
노이즈 값을 샘플링할 가우스 분포(Gaussian distribution)의 평균값(mean), 표준편차(std)를 설정할 수 있다.

노이즈 값은 각 pixel 마다 독립적으로 샘플링된다. 즉, 노이즈 값은 픽셀의 각각의 color channel 마다 독립적으로 추가된다.
    카메라의 한 픽셀은 R, G, B의 조합으로 이루어져 있다.
    노이즈가 각 채널에 따로 들어가기 때문에, 노이즈가 심해지면 화면에 알록달록한 아지랑이가 피어오르게 된다.
노이즈를 추가한 후, color channel의 결과값은 0과 1 사이에 놓인다.
    시뮬레이션 내부적으로 카메라의 색상 값은 0~1 범위 실수로 연산된다.
    만약 원래 픽셀의 R 값이 0.9인데, 가우시안 노이즈 0.2가 더해지면 Gazebo가 이를 1.0으로 잘라준다(Clipping).
std 값을 내리면, 좋은 카메라의 결과물을 보는 듯 한 결과를 얻을 수 있다.
    - std 값이 크다: Gaussian 그래프가 넓게 퍼진다. 원래 나와야 할 색상과 동떨어진 색상(큰 노이즈)가 찍힐 확률이 높다는 것을 의미한다.
    - std 값이 직다: 그래프가 평균을 향해 뾰족하게 모인다. 오차가 발생하더라도 미세한 오차만 발생한다.

- IMU(관성 측정 장치)
IMU는 가속도와 각속도를 측정한다.
    가속도(Linear Accel): X, Y, Z축 방향으로 얼마나 빠르게 가속하는지(m/s^2)
    각속도(Angular Rate): 로봇이 각 축을 기준으로 얼마나 빠르게 회전하는지(rad/s)

Gazebo는 이 두가지 물리량 각각에 Noise와 Bias라는 두 종류의 방해 요소(Disturbance)를 입력할 수 있게 해두었다. -> 총 4 세트의 파라미터가 필요하다.
    Accel Noise (가속도 노이즈)
    Rate Noise (각속도 노이즈)
    Accel Bias (가속도 편향 오차)
    Rate Bias (각속도 편향 오차)

Noise와 Bias : 현실의 IMU 센서 데이터에는 두 가지가 동시에 작용한다.
    - Noise (Gaussian노이즈)
        - 센서 칩 내부의 미세한 전기적 신호 흔들림 때문에 발생
        - 정답을 중심으로 위아래로 매우 빠르게 떨리는 형태. 평균이 0이기 때문에, 데이터를 오랫동안 평균 내거나 필터를 거치면 어느 정도 상쇄되어 정답에 가까워진다.
    - Bias (편향)
        - 센서의 온도 변화, 납땜 상태, 부품 노화 등으로 인해 발생하는 치명적인 오차
        - 로봇이 가만히 멈춰 있어도 항상 정해진 상수값(Bias)을 출력한다.
Noise가 로봇 제어에 미치는 영향(누적 오차) - 현실 IMU의 가장 큰 난제
    - 가속도를 한번 적분하면 속도, 두 번 적분하면 이동 거리(위치)가 된다.
    - 각속도를 한번 적분하면 로봇의 회전 각도(자세)가 된다.
    따라서, 오차가 누적된다면 적분 값이 커져서 실제로는 로봇이 가만히 있는데 로봇이 움직이고 있다고 착각을 하게 되는 드리프트(Drift) 현상이 발생할 수 있다.

Gazebo가 어떤 수학적/논리적 방식으로 IMU에 Noise와 Bias를 더하는지에 대한 핵심 메커니즘은 다음과 같다.
1. Noise: 매 순간 새로 계산되어 더해지는 값
    - 동작 방식: 시뮬레이션이 굴러가는 매 순간, X, Y, Z축 각각에 대해 새로운 가우시안 노이즈 값을 뽑아낸다.
    - X, Y, Z 축의 값은 독립적으로 샘플링(랜덤하게 결정)된다.
    - 각속도(Rates) 용 가우시안 분포의 평균, 표준편차와 가속도(Accel) 용 가우시안 분포의 평균, 표준편차만 설정하면 된다. -> 이 분포를 가지고 X, Y, Z 축에 동일하게 적용한다.
2. Bias: 시뮬레이션이 시작할 때 딱 한번 결정되는 고정 값
    - 시뮬레이션이 켜지는 맨 처음 순간에 딱 한번 가우시안 분포에서 값을 뽑아낸다.
    - 한번 뽑힌 Bias 값은 시뮬레이션이 끝날 때까지 변하지 않는 고정값(fixed)이 된다.

    Gazebo가 가우시안 노이즈 값을 샘플링하는 방법
        1. Bias의 평균을 0.1로 설정한 경우를 가정한다.
        2. 가우시안 분포 0.1 근처에서 값을 샘플링한다.
        3. 그 후, 50% 확률(equal probability)로 마이너스 부호를 붙일지 말지 결정
        현실에서 오차가 어느 방향(+/-) 일지 알 수 없기에, 50% 확률로 결정해 준다.

IMU 센서 데이터의 최종 형태: 
Output = True Value (정답) + Bias (처음 켤 때 고정된 오차) + Noise(매 순간 바르르 떠는 오차)

### Plugins 내용 의미 공부
```xml
<plugin filename="libgazebo_ros_imu_sensor.so" name="gazebo_ros_imu_sensor">
    <ros>
      <namespace>/robot</namespace>
      <remapping>~/out:=imu_data</remapping> 
    </ros>
    <frame_name>imu_link</frame_name>
  </plugin>
```
- 내 컴퓨터 어딘가에 저장되어 있는 filename 이름의 빌드된 라이브러리를 찾아서 실행해라
    - 파이썬의 "import ~ as ..." 와 같은 기능 및 코드로 이해하면 될 듯
- <ros> 태그 내부와 <frame_name>은 노이즈가 섞인 가짜 IMU 데이터를 어떤 이름의 채널로, 어떤 형태의 규격에 맞춰 보낼 것인가를 결정하는 구체적인 배달 주소지 설정
    - namespace : 해당 로봇에서 얻는 모든 데이터의 주소 앞에 접두어를 붙여라!
    - remapping : 플러그인이 내보내는 기본 채널 이름을, 내가 원하는 이름으로 바꿔서 내보내라! (닉네임 변경 링크 역할)
    - framename : 어느 물리적 위치 기준으로 측정된 값인지 데이터 헤더에 박아서 보내라!
