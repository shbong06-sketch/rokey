- **목적:** 벤츠 차량의 옵션 조합을 기반으로 안전 테스트 시간(y)을 예측하기 위한 기본 데이터 구조 파악

## 0. 제조 도메인 지식 및 배경 (Domain Knowledge)

본 프로젝트의 타겟 변수인 안전 테스트 시간($y$)을 깊이 이해하기 위해 자동차 최종 검사 공정의 특성을 다음과 같이 정리하고 분석의 가정으로 삼는다.
이를 통해 데이터 각 Column이 가진 의미에 대해서도 추정할 수 있도록 한다.

### 자동차 공정의 순서
- 자동차 공정은 크게 **프레스** ➡️ **차체** ➡️ **도장** ➡️ **의장(조립)** ➡️ **검사** 순으로 진행됨.

- 제조 공저에서 이루어지는 검사 과정(종류)

    - 휠 얼라인먼트 및 브레이크 테스트: 바퀴 정렬과 제동 장치 확인.

    - 드럼 테스트 (환경/동력 테스트): 차량을 롤러 위에 올리고 실제로 시속 100km 넘게 달리며 엔진, 변속기, 구동계 이상 여부 확인.

    - 누수 테스트 (Shower Test): 고압의 물을 뿌려 차 내부로 물이 새는지 확인.

    - 전장 시스템 및 ADAS 검사: 카메라, 레이더, 내비게이션, 센서류의 소프트웨어 및 하드웨어 통신 확인.

#### 차량마다 검사 시간이 다른 이유
- 일반적으로 공장에서는 한 종류의 제품만 만들지 않고, 같은 라인에서 여러 종류의 제품의 옵션이나 종류가 섞여 지나가는 혼류 생산(Mixed-Model Production) 방식을 택함.

- 예시

    - 옵션의 차이: 어떤 차량은 자율주행(ADAS) 옵션이 있고, 어떤 차량은 일반 크루즈 컨트롤만 있음. ADAS 옵션이 있는 차량은 센서 보정(Calibration)과 통신 테스트를 추가로 해야 하므로 검사 시간이 길어짐.

    - 구동 방식의 차이: 2륜 구동 차량은 두 바퀴만 굴려보면 되지만, 4륜 구동(AWD) 차량은 네 바퀴를 모두 제어하는 특수 드럼 테스트가 필요해 공정 시간이 추가.

#### 변수: Bottleneck(병목)과 이상치(Outlier)
- 생산 라인은 컨베이어 벨트처럼 일정한 속도(Tact Time)로 움직임.
- 유독 어떤 차량 하나가 검사장에서 오랜 시간 붙잡혀 있다면(데이터의 최댓값처럼), 뒤에 오던 차량들이 줄줄이 대기해야 하는 병목 현상(Bottleneck)이 발생.

- 일반적으로 발생하는 상황
    - 복잡한 최고급 풀옵션 차량: 검사해야 할 전장 부품과 패키지가 너무 많아서 물리적인 절대 시간이 오래 걸리는 경우.

    - 불량 발생 및 재작업(Rework): 테스트 도중 센서 에러나 조립 불량이 감지되어, 검사원이 차를 세워두고 디버깅을 하거나 값을 재세팅하느라 시간이 늘어진 경우.

### ① 의장 공정 후 최종 검사 (Final Inspection)
- 차량 조립(의장) 완료 후, 휠 얼라인먼트, 드럼 테스트, 누수 테스트, 전장/ADAS 시스템 검사 등이 연속적으로 수행됨.
- 우리가 예측해야 하는 $y$값은 이 개별 검사 시간들의 총합(초)임.

### ② 혼류 생산(Mixed-Model Production)과 옵션 변수
- 현대 자동차 공장은 한 라인에서 다양한 차종과 옵션을 동시에 생산함.
- 익명화된 이진 변수(`X10`~`X385`)들은 차량에 장착된 **특정 세부 옵션/기능의 유무(0 또는 1)**를 의미할 가능성이 매우 높음. 
- ADAS, 4륜 구동 등 고부가가치/복잡한 옵션이 추가될수록 검사 항목이 늘어나 $y$값(시간)이 증가하는 경향을 보일 것으로 가정함.

### ③ 공정 병목(Bottleneck)과 재작업(Rework)
- 데이터의 최댓값(265초 등)처럼 유독 긴 검사 시간을 가진 이상치(Outlier)가 존재함.
- 이는 단순히 옵션이 많아서일 수도 있지만, **검사 중 불량/에러가 발생하여 현장 작업자가 디버깅 및 재작업(Rework)**을 하느라 라인이 지연된 케이스일 수 있음. 
- 따라서 이 이상치들을 어떻게 처리하느냐가 모델의 일반화 성능의 핵심 열쇠가 될 것임.

## 1. 데이터 개요 (Data Shape)
- **Train 데이터 수:** 4,209 행(Row) / 378 열(Column)
  - 행에 비해 열(Feature)의 수가 압도적으로 많은 **고차원 데이터**.
  - 차원의 저주를 방지하기 위해 추후 변수 선택(Feature Selection)이나 차원 축소(PCA) 고려 필요.
- **결측치 (Missing Value):** 전체 데이터셋에 결측치 없음 (매우 깨끗한 상태).

## 2. 변수 특성 분석 (Feature Types)
데이터를 크게 두 가지 유형으로 분류할 수 있음:

1. **범주형 변수 (Categorical Variables):** `X0`, `X1`, `X2`, `X3`, `X4`, `X5`, `X6`, `X8` (총 8개)
   - 문자열(텍스트) 데이터
   - 머신러닝에 입력하기 전 원-핫 인코딩(One-Hot Encoding)이나 타겟 인코딩(Target Encoding) 필수.
   - *메모: 특히 `X0`는 종류가 많아 타겟(y)에 큰 영향을 줄 것으로 예상됨.*
   
   + `x0` : 타겟의 수 47개. 변별력이 가장 큰 변수
   + `x3`, `x4` : 타겟의 수가 7개/4개로 적은 변수

2. **이진 변수 (Binary Variables):** `X10`부터 `X385`까지의 변수들
   - 수백 개의 변수가 전부 `0` 또는 `1`로만 이루어짐 (특정 세부 옵션의 유무로 추정).
   - *주의:* 4,209개 행 전체가 모두 `0`이거나 모두 `1`이어서 변별력이 없는 '분산이 0인 변수'들이 존재함. 전처리 단계에서 제거 필요.
   + `X11`, `X93`, `X107`, `X233`, `X235`, `X268`, `X289`, `X290`, `X293`, `X297`, `X330`, `X347` : 분산이 0인 데이터 -> 모든 행의 값이 동일하여 변별력이 없는 변수

## 3. 타겟 변수 분석 (Target Variable: y)
모델이 예측해야 하는 정답인 `y`(테스트 시간, 초)의 통계적 특성:

- **평균(Mean):** 약 100.6초
- **중앙값(50%):** 약 99.1초 (평균과 중앙값이 비슷하여 비교적 안정적인 분포)
- **최솟값(Min):** 72.1초 / **최댓값(Max):** 265.3초
- **💡 인사이트 (이상치 발견):** - 75% 지점이 109초인데 최댓값이 265초인 것은 유독 검사 시간이 오래 걸린 특이 차량(Outlier)이 존재한다는 뜻.
  - 이 이상치들을 모델에 그대로 학습시키면 예측력이 망가질 수 있어 보임.

### 이상치 처리 전략
1. 로그 변환 (Log Transformation)
- 타겟 변수가 한쪽으로 치우쳐 있고(Right-skewed) 큰 이상치가 있을 때 가장 표준적으로 쓰는 방법
    + 원리: 파이썬의 np.log1p(y)를 사용해 값에 로그를 취함. 큰 값일수록 압축되는 효과가 강해서, 260초처럼 멀리 떨어져 있던 이상치들이 다른 데이터들 근처로 이쁘게 모이게 됨.

    + 장점: 데이터를 강제로 삭제하지 않기 때문에 데이터 손실이 없음. 모델이 학습하기 가장 좋은 예쁜 정규분포 모양으로 만들어 줌.

    + 주의점: 모델이 예측한 결과도 로그 값이므로, 최종 제출할 때는 다시 지수 함수(np.expm1())를 취해 원래 초(second) 단위로 되돌려놓아야 함.

2. 박스 플롯 기준(IQR)으로 상한선 자르기 (Clipping / Capping)
- 수학적 공식인 IQR(Interquartile Range)을 사용해 정상 범위를 벗어나는 아주 극단적인 값들을 특정 최댓값으로 강제로 고정시키는 방법
    + 원리: $Q3 + 1.5 \times IQR$ 공식을 구했더니 기준이 130초가 나왔다면, 130초를 넘어가는 모든 데이터(예: 150초, 200초)를 전부 똑같이 130초로 바꿔버리는 방법.

    + 도메인 관점 해석: "아무리 불량이 나서 재작업(Rework)을 하더라도, 생산 관리 프로세스상 최대 검사 시간 마지노선은 130초로 제한한다"라고 공학적 한계선을 긋는 것과 같음.

    + 장점: 극단적인 이상치 때문에 모델의 예측선이 위로 확 꺾이는 현상을 방지함.

3. 이상치 데이터 아예 삭제 (Removal)
- 기준을 벗어나는 행(Row)을 데이터셋에서 아예 지워버리는 방법

    + 원리: 이상치를 데이터 입력 오류이거나 너무 특이한 케이스로 판단해 학습에서 제외시킴. train_df = train_df[train_df['y'] < 150] 형태로 드롭.

    + 주의점: 데이터 손실이 발생하고, 모델이 이상치를 예측하는 방법을 배우지 못함.

 4. 이상치 유무를 알려주는 '파생 변수' 추가
 - 이상치를 그대로 두되, 모델에게 힌트를 주는 고급 기법.

    + 원리: is_outlier라는 컬럼을 새로 만들어서, $y$값이 130초 이상인 행에는 1, 정상 차량에는 0을 부여함.
    
    + 도메인 관점 해석: 모델에게 *"이상치에 대해, 일반적인 옵션 조합이 아니라 공정 상 에러(Rework)가 난 특이 케이스*라고 라벨을 붙여주는 것. 모델이 이상치 데이터의 패턴을 따로 분리해서 학습할 수 있도록 도와줌.

## 🚀 전처리 단계
### 정답 `y` 변수의 히스토그램 그려서 분포 시각화
- 원본 vs 로그 변환 비교
    - 원본: 우편향(Right-skewed) 분포
    - 로그 변환: 우측으로 멀리 떨어져 있던 데이터들이 중심 축(4.4 ~ 4.8) 근처로 강하게 당겨져 옴.
    - 정규분포는 아니지만, 봉우리가 여러 개 있는 형태(Multimodal)를 유지한 채 데이터의 스케일(범위)이 압축됨.

- 이상치 기준
    - Boxplot 그래프 상으로 확인했을 때, 대략 130~140 부근이 적당해보임.
    - **IQR(Interquartile Range, 사분위간 범위) 규칙**
        - 데이터의 실제 분포가 얼마나 퍼져있는지를 기준으로 삼아 정상 범위를 계산
        - 하위 25% 지점을 $Q_1$, 75% 지점을 $Q_3$라고 함.
        - $$\text{IQR} = Q_3 - Q_1$$
        - $$\text{정상 범위 상한선} = Q_3 + (1.5 \times \text{IQR})$$

        + 이상치 기준 활용 방안
        1. 파생 변수로 모델에 직접 입력 (train 세트 한정)
        2. [성능 모니터링] 훈련 과정에서 오차 검증용
        3. [비즈니스 의사결정] 생산 관리용 알람 시스템 (실무 핵심)

- 이상치 여부 확인 라벨(`is_outlier`) 생성
    - is_outlier 컬럼 추가 -> upper_bound 기준으로 판단


### 분산이 0인(모든 값이 똑같은) 무의미한 이진 변수 찾아내서 드롭(Drop)하기



### 범주형 변수들(`X0`~`X8`)의 고유값 개수(Unique Count) 확인하기
