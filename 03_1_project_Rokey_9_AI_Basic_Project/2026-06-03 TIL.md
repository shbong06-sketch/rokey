## 3. 주요 설정 단계 및 명령어
### 단계 1: v1 모델
- weak data augmentation
    - v2.RandomResizedCrop((92, 92), scale=(0.8, 1.0))
    - v2.RandomHorizontalFlip()
- 모델 구조 개선
    - resnet18
    - dropout(0.5)가 가장 효과적
- learning rate scheduler 사용
    - 과적합 방지를 위함(train_acc, test_acc의 gap 감소)
    - T_max=30(num_epoch와 동일)
- optimizer 변경
    - SGD with momentum 낮은 정확도
    - AdamW (weight decay=0.05) 더 높은 정확도, 비슷한 수준의 gap
- label smoothing = 0.05
    - 과적합 방지를 위함(gap 소폭 감소)

### 단계 2: v2 모델
- medium augmentation
    - v2.RandAugment(num_ops=2, magnitude=9) : 정확도 및 gap 개선
        - Rotate, ShearX/Y, TranslateX/Y, Brightness, Contrast, Color, Sharpness, Posterize, Solarize, Equalize, AutoContrast 등의 변환 중 랜덤하게 선택
    - v2.RandomErasing(p=0.5, scale=(0.02, 0.33), ratio=(0.3, 3.3), value=0, inplace=False)
    - RandAugment의 인수 조정 >> num_ops=3 / magnitude=7 : 더 낮은 정확도
- 현재 상황 분석
    - 최고 성능 : 73.2%, gap 11.0667
    - 더이상 hyperparameter tunning으로 유의미한 성능 개선을 얻지 못한다. (정체 구간)
- 다음 단계 구상
    - 모델 한계 : ResNet18
    - 이미 강한 augmentation이 가해지고 있는 듯 하다.
        - RandomErasing했을 때, 오히려 underfitting 되는 듯한 모습을 보이기도 함.
    - cutmix vs mixup : 데이터 분포 자체 변경 시도해보기.
    - epoch 증가 : 30 >> 50 (with early stopping)


### 단계 3: v3 모델
- Cutmix or Mixup 적용
    - 오히려 train acc 값이 감소 >> underfitting 문제 발생
    - 지나치게 강한 augmentation이 과하게 적용된 듯
    - Cutmix or Mixup 제거
        - Cutmix와 Mixup은 학습 데이터 크기가 충분히 큰 경우에 효과적인 모델

### 단계 4: v4 모델
- 모델 변경 : ResNet34
    - 오히려 정확도 감소
        - 입력 이미지 크기가 92*92라 깊은 모델에서 오히려 과적합될 위험이 존재
        - 데이터augmentation이 강하게 들어간 영향도 있는 듯하다.
            - RandAugment의 magnitude 줄여보기 9 >> 7(정확도 증가 경향) >> 5 (큰 차이 없다.)
        - magnitude를 조정해봤지만, 오히려 test 정확도가 4% 가량 떨어짐.
    - 이미지 크기가 작은 데이터에서는 ResNet18이 최적
    - epoch 수 증가(50) : 비교적 깊은 모델이니, 학습 반복 수 증가


## 트러블슈팅

### 에러 발생 및 현상
- 상황: 실험 과정에서 RandomAugment만 추가했는데, 모델의 Validation 정확도가 갑자기 지나치게 높은 결과가 나옴(+25% 수준)
- 에러 메시지: val_acc : 71.6% >> 96.2%
- 해결 방법 : 
    - random seed 값 바꾸고, 실험 진행 >> 73.8%로 줄어들었다.
    - 1차 원인 분석 : test-val 분할 과정에서 Data Leakage이 발생한 것 같다.
        - 
    - 2차 원인 분석 : test-val 분할 과정에 별도의 Seed 고정이 없어서 아주 낮은 확률로 쉬운 val 데이터가 들어온 것 같다.
        - 시드 고정 실행
        \`\`\`python
        generator = torch.Generator().manual_seed(42)
        \`\`\`
    - 3차 원인 분석 : 실험 단계에서 모델을 새롭게 생성하지 않았다.
        - state를 지닌 객체를 실험 시마다 재실행하지 않아, 독립 비교 실험으로 보기 어려움.
        - 마치 epoch을 늘리며 학습을 이어서 한 것과 같은 효과가 발생한다.
- 결과 및 배운 점
    - 각 실험을 진행할 때, 모델을 새롭게 생성하지 않았다. >> 독립 비교 실험으로 보기 어려움.
    - statefull한 객체는 매 실험 시마다 재실행이 필요(그렇지 않으면, epoch를 늘려가며 반복하는 효과가 발생)
    - 실험 진행 시 프로토콜
        1. Seed 고정
        2. random_split
        3. 모델 새로 생성
        4. Optimizer 새로 생성
        5. Scheduler 새로 생성
        6. 학습
